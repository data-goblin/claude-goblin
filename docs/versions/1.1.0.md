# Version 1.1.0 - Windows Startup Fix & Limits Removal

Released: July 14, 2026

## Overview

Bugfix release: the CLI now starts on Windows, `ccg export` works again, and
the long-disabled limits tracking feature is removed.

## Fixes

### Windows startup crash (#4)

`ccg` failed to start on Windows with:

```
ModuleNotFoundError: No module named 'termios'
```

`src/commands/limits.py` imported the POSIX-only `pty` module at module
level, and every command imported it transitively, so the whole CLI was
unusable on Windows. The module is gone entirely in this release.

### `ccg export` crash

`ccg export` crashed on startup with a `TypeError` while evaluating the
`datetime.date | None` type annotations: the module imported the `datetime`
class, shadowing the `datetime` module, so `datetime.date` resolved to a
method instead of a type. Annotations now use the real `date` type.

### Stale `--version`

`ccg --version` reported a hardcoded `0.2.0`. It now reports the actual
package version.

## Removed: limits tracking

Limits tracking worked by scraping `claude /usage` through a pseudo-terminal
and has been non-functional since Claude Code changed its `/usage` output
format. Rather than keep shipping dead code, the feature is removed:

- `ccg limits` and `ccg status-bar` commands
- The `/usage` pty scraper and macOS menu bar app
- Limits panels in the `ccg usage` dashboard
- `ccg export --show tokens|limits|both` flag; export always shows the token
  heatmap now
- The `tracking_mode` config key (token tracking is always on)

Run `claude /usage` directly to view your limits.

### Data preservation

The `limits_snapshots` table and its sync paths (Quack, OneLake, migration)
are retained, so any historical limits data you captured stays in your local
and remote databases.
